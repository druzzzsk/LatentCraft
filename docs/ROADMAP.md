# LatentCraft — Roadmap

## Концепция

**LatentCraft** — проект по оптимизации молекулярных свойств через навигацию в латентном пространстве.

**Идея:** дискретные химические структуры (SMILES / графы) кодируются в непрерывное латентное пространство с помощью VAE. В этом пространстве можно двигаться в направлении улучшения целевых свойств (logP, QED, SA score) и декодировать результат обратно в валидную молекулу.

**Стратегия проекта:** не обучать архитектуры с нуля, а взять проверенные open-source реализации молекулярных VAE, воспроизвести их результаты, а затем систематически сравнить разные методы оптимизации в их латентных пространствах. На основе этого сравнения — предложить и реализовать собственные улучшения.

**Базовые репозитории:**

| Кодовое имя | Репо | Представление | Статус pretrained |
|---|---|---|---|
| SMILES-VAE | `aspuru-guzik-group/chemical_vae` | SMILES (one-hot) | Переобучить на ZINC250K |
| JT-VAE | `wengong-jin/icml18-jtnn` | Junction tree | Есть в репо (старый PyTorch, возможен патчинг) |
| SELFIES-VAE | `aspuru-guzik-group/selfies` + адаптация arch. | SELFIES строки | Обучить самому (адаптация SMILES-VAE) |
| HierVAE *(опц.)* | `wengong-jin/hgraph2graph` | Иерархический граф | Есть в репо (аналогичные проблемы совместимости) |

**Методы оптимизации для сравнения:**

- Gradient ascent (через property predictor)
- Bayesian Optimization (GP-based, BoTorch)
- Evolutionary strategies (CMA-ES)

## Фаза 1 — Evaluation framework

> Фреймворк оценки нужно построить **до** экспериментов, чтобы все модели и оптимизаторы оценивались единообразно.

### 1.1 Метрики генерации молекул

Реализовать модуль `evaluation/metrics.py` с расчётом стандартных метрик:

| Метрика | Что измеряет |
|---|---|
| **Validity** | % декодированных SMILES, прошедших RDKit-парсинг |
| **Uniqueness** | % уникальных среди валидных |
| **Novelty** | % не встречающихся в обучающей выборке |
| **Diversity** | средний внутри-выборочный Tanimoto distance |
| **Reconstruction accuracy** | % точно восстановленных из латентного вектора |

### 1.2 Метрики оптимизации свойств

Реализовать метрики, специфичные для задачи оптимизации:

| Метрика | Что измеряет |
|---|---|
| **Property improvement** | Δ целевого свойства (logP, QED, SA) относительно seed |
| **Success rate** | % молекул с улучшением > порога |
| **Similarity to seed** | Tanimoto (Morgan FP) между seed и оптимизированной молекулой |
| **Pareto efficiency** | доля решений на Pareto-фронте (improvement vs. similarity) |

### 1.3 Визуализация и отчёты

Скрипт / notebook для автоматической генерации сравнительных таблиц и графиков:
- Таблица `модель × оптимизатор × метрика`
- Scatter-plot: property improvement vs. structural similarity
- Латентное пространство (t-SNE / UMAP), раскрашенное по свойству
- Pareto-фронт для multi-objective случая

## Фаза 4 — Вторая и третья модели

> **Контекст выбора моделей.** Каждая модель использует принципиально разное молекулярное представление, что напрямую влияет на структуру латентного пространства и эффективность оптимизаторов. Цель — не просто запустить три модели, а выяснить, какое представление даёт более гладкое и управляемое латентное пространство.

### 4.1 JT-VAE (фрагментное дерево)

**Репо:** `wengong-jin/icml18-jtnn` — ICML 2018, ~554 stars  
**Представление:** молекула как дерево из химических фрагментов (rings + bonds). Декодер сначала строит дерево фрагментов, затем собирает атомы. Validity близка к 100%.  
**Особенность по сравнению с SMILES-VAE:** латентное пространство учитывает химическую структуру явно, а не косвенно через символьные паттерны.

**Шаги интеграции:**

1. Клонировать репо, установить зависимости. Код под старый PyTorch — потребуется патчинг: заменить `Variable(...)` на прямые тензоры, проверить устаревшие API (`volatile`, `requires_grad` в конструкторе).
2. Попытаться загрузить pretrained чекпоинт из репо (обучен на ZINC250K). Если не загружается из-за несовместимости — переобучить с нуля на ZINC250K (~6-10 часов на GPU, процедура задокументирована в репо).
3. Обернуть в `BaseVAE` (encode, decode, sample, reconstruct).
4. Обучить property predictor (logP) поверх латентных векторов JT-VAE.
5. Прогнать все 3 оптимизатора, зафиксировать метрики из фазы 1.

### 4.2 SELFIES-VAE (гарантированная валидность)

**Библиотека:** `aspuru-guzik-group/selfies`  
**Представление:** SELFILES-алфавит гарантирует, что любая последовательность символов декодируется в валидную молекулу. Это устраняет проблему невалидных зон в латентном пространстве — оптимизатор может двигаться в любом направлении без риска выйти в "мусор".  
**Особенность:** нет готового pretrained — архитектуру нужно обучать самому, но это несложно.

**Шаги интеграции:**

1. Конвертировать ZINC250K из SMILES в SELFIES: `selfies.encoder(smiles)` для каждой молекулы, сохранить датасет.
2. Адаптировать архитектуру SMILES-VAE из фазы 2: заменить токенизатор и словарь на SELFIES-алфавит, остальная архитектура (encoder GRU/Conv, reparameterization, decoder) остаётся той же.
3. Обучить VAE на SELFIES ZINC250K (~то же время, что и SMILES-VAE).
4. Обернуть в `BaseVAE`. Проверить reconstruction accuracy и validity.
5. Обучить property predictor (logP), прогнать 3 оптимизатора, зафиксировать метрики.

### 4.3 (Опционально) HierVAE — иерархический граф

**Репо:** `wengong-jin/hgraph2graph`  
**Представление:** молекула кодируется на двух уровнях — крупные фрагменты (scaffold) и атомарные детали внутри каждого фрагмента. Теоретически позволяет управлять оптимизацией на разных уровнях абстракции.  
**Почему опционально:** тот же автор, что JT-VAE — аналогичные проблемы с совместимостью кода. Интеграция сложнее, и для gradient ascent иерархический декодер создаёт дополнительные сложности.

**Если берёмся:**

1. Клонировать репо, пропатчить PyTorch API (аналогично JT-VAE).
2. Попытаться загрузить pretrained чекпоинт на ChEMBL (есть в репо). При неудаче — переобучить на ZINC250K.
3. Обернуть в `BaseVAE`, прогнать оптимизаторы, добавить в сравнительную таблицу.

## Фаза 5 — Собственные улучшения

> Конкретные направления определяются по итогам фаз 3-4. Ниже — кандидаты.

### 5.1 Multi-objective оптимизация

Одновременная оптимизация нескольких свойств (например, QED + SA + similarity). Построение Pareto-фронта. Сравнить с single-objective подходами.

### 5.2 Constrained optimization

Оптимизация с ограничением: сохранять молекулярный scaffold или подструктуру seed-молекулы. Реализовать как штраф или как hard constraint.

### 5.3 Uncertainty-aware navigation

Использовать неопределённость property predictor (ensemble / MC dropout) для того, чтобы оставаться в регионах латентного пространства с надёжными предсказаниями.

На основе выводов — предложить изменения в архитектуре или loss-функции (например, property-conditioned prior, disentangled latent space, weighted ELBO).
